## 🔧 规则文件命名

### 编号规则文件
- **格式**: `NN-中文名称.md` 或 `NN-descriptive-name.md`
- **示例**: `01-文件操作规范.md`, `03-文件命名规范.md`
- **编号**: 00-99，两位数字
- **名称**: 小写英文（连字符分隔）或中文

### 项目配置文件
- `project-config.md` - 项目特定配置
- `project-config.example.md` - 配置模板

## 📝 文档命名规范

### 1. 记录类文档（推荐日期在前）⭐

**核心原则**: **日期在前，主题在后**

#### 基本格式
```
YYYY-MM-DD_[类型]记录.md
YYYY-MM-DD_[主题]报告.md
YYYY-MM-DD_[描述].md
```

#### 为什么日期要在前？
- ✅ **自动时间排序** - 文件系统按文件名排序时，自动按日期顺序
- ✅ **时间线清晰** - 一眼看出记录的时间顺序
- ✅ **查找方便** - 想找某天的记录，直接按日期查找
- ✅ **符合习惯** - Git日志、系统日志都是日期在前
- ✅ **避免混乱** - 同类记录自动聚合在一起

#### 示例对比
```markdown
✅ 正确（日期在前）：
2025-11-20_项目初始化记录.md
2025-11-21_开发环境搭建.md
2025-11-25_功能开发记录.md         自动按日期排序
2025-11-26_测试结果报告.md
2025-11-27_部署准备记录.md

❌ 错误（主题在前）：
功能开发记录_2025-11-25.md        按"功"字排序，时间顺序丢失
测试结果报告_2025-11-26.md        无法快速定位时间
```

#### 适用场景
- 📅 日常开发记录
- 📋 会议记录/会议纪要
- 📊 测试报告
- 📝 工作日志
- 🔄 变更记录
- 🐛 问题排查记录
- 👥 代码审查记录

#### 文件组织规则 ⭐

**强制要求**：同一天的记录文档必须放在对应日期的子目录中

**目录结构**：
```
项目文档/2-开发记录/
├── YYYY-MM-DD/                    ← 日期目录
│   ├── YYYY-MM-DD_报告A.md       ← 该日期的所有文档
│   ├── YYYY-MM-DD_记录B.md
│   └── YYYY-MM-DD_分析C.md
├── YYYY-MM-DD/                    ← 另一个日期
│   └── YYYY-MM-DD_文档.md
└── README.md
```

**正确示例**：
```
✅ 项目文档/2-开发记录/2025-11-30/2025-11-30_测试覆盖率验证报告.md
✅ 项目文档/2-开发记录/2025-11-25/2025-11-25_目录清理记录.md
```

**错误示例**：
```
❌ 项目文档/2-开发记录/2025-11-30_测试覆盖率验证报告.md    （应该在 2025-11-30/ 子目录内）
❌ 项目文档/2025-11-30_某个报告.md                      （应该在 2-开发记录/2025-11-30/ 内）
```

**为什么要这样组织？**
- ✅ 避免主目录文件过多
- ✅ 按日期分组，便于管理
- ✅ 每天的工作内容独立归档
- ✅ 方便批量操作某一天的文档

### 2. 时间戳使用规范 ⏰

#### 同一天有多个记录时

**格式**: `YYYY-MM-DD_HH-MM_[描述].md`

**示例**:
```markdown
✅ 添加时间戳：
2025-11-25_10-30_晨会记录.md
2025-11-25_15-45_代码审查记录.md
2025-11-25_18-00_部署总结.md

✅ 或添加主题细分：
2025-11-25_用户模块开发记录.md
2025-11-25_数据库优化记录.md
2025-11-25_前端界面调整记录.md

✅ 或添加序号：
2025-11-25_开发记录_01.md
2025-11-25_开发记录_02.md
```

### 3. 不带日期的文档

#### 规范类
- **格式**: `[主题]规范.md`
- **示例**: `文件操作规范.md`, `API设计规范.md`, `代码审查规范.md`

#### 说明类
- **格式**: `[主题]说明.md`
- **示例**: `部署说明.md`, `配置说明.md`, `接口说明.md`

#### 指南类
- **格式**: `[主题]指南.md`
- **示例**: `新手指南.md`, `开发指南.md`, `故障排查指南.md`

#### 文档类
- **格式**: `[主题]文档.md`
- **示例**: `API文档.md`, `架构文档.md`, `需求文档.md`

---

### 4. 版本化文档

**格式**: `[主题]_v[版本号].md`

**示例**:
```markdown
架构设计_v1.0.md
架构设计_v2.0.md
API文档_v3.2.md
```

**或使用版本目录**:
```
API文档/
 v1.0.md
 v2.0.md
 v3.0.md
```

## 📅 日期格式规范

### 标准格式

| 场景 | 格式 | 示例 | 说明 |
|------|------|------|------|
| 日期 | `YYYY-MM-DD` | `2025-11-25` | ISO 8601标准 |
| 日期+时间 | `YYYY-MM-DD_HH-MM` | `2025-11-25_14-30` | 使用 `_` 和 `-` 分隔 |
| 日期+时间+秒 | `YYYY-MM-DD_HH-MM-SS` | `2025-11-25_14-30-45` | 精确到秒 |

### 分隔符规范
- ✅ 日期内部使用 `-`：`2025-11-25`
- ✅ 日期与描述使用 `_`：`2025-11-25_描述`
- ✅ 时间内部使用 `-`：`14-30-45`
- ❌ 不使用 `.` 或其他符号

### 示例
```markdown
✅ 正确格式：
2025-11-25_开发记录.md
2025-11-25_14-30_会议记录.md
2025-11-25_14-30-45_测试报告.md

❌ 错误格式：
20251125_开发记录.md              # 日期没有分隔符
2025.11.25_开发记录.md            # 使用了 .
2025_11_25_开发记录.md            # 分隔符不统一
2025-11-25-开发记录.md            # 日期与描述使用了 -
```

## 🔴 AI文件命名前强制检查

**在创建任何文件之前**，AI必须检查文件名：

 **检查1**: 禁止的模糊名称
   - 文件名是文档.md、说明.md、笔记.md、记录.md吗？
   - 如果是  ❌ 禁止！必须具体化

 **检查2**: 禁止的临时名称
   - 文件名包含"临时"、"test"、"测试"、"TODO"、"草稿"吗？
   - 如果是  ❌ 禁止！必须使用正式名称

 **检查3**: 包含空格
   - 文件名有空格吗（如我的 文档.md）？
   - 如果有  ❌ 禁止！使用下划线或连字符

 **检查4**: 随意命名
   - 文件名是aa.md、xxx、111这类吗？
   - 如果是  ❌ 禁止！必须有意义

**如果发现文件名违规**：
1. ⏸️ 立即停止创建
2. 💬 提示用户："文件名不符合规范，请使用清晰具体的名称"
3. ✏️ 建议正确的命名

**AI违规记录**：
- 如果创建了禁止文件名  立即删除重命名

## 🚫 禁止使用的名称

### ❌ 绝对禁止

#### 模糊名称
```markdown
❌ 文档.md
❌ 说明.md
❌ 笔记.md
❌ 记录.md
❌ README2.md
```

#### 临时名称
```markdown
❌ 临时.md
❌ 临时笔记.md
❌ test.md
❌ 测试.md
❌ 新建文档.md
❌ TODO.md
❌ 草稿.md
```

#### 包含空格
```markdown
❌ 我的 文档.md
❌ 项目 说明.md
❌ 开发 记录.md
```

#### 随意命名
```markdown
❌ aaa.md
❌ xxx.py
❌ temp123.js
❌ 111.md
❌ 未命名.md
```

#### 不规范的日期
```markdown
❌ 20251125_记录.md           # 日期没分隔符
❌ 2025.11.25_记录.md         # 使用了 .
❌ 11-25-2025_记录.md         # 美式日期格式
```

## 🔄 重命名原则

### 遇到模糊文件名时

**流程**:
1. ⏸️ 停止操作
2. 💬 询问用户明确名称
3. 📝 使用清晰具体的命名
4. 🔗 更新所有引用

**示例**:
```markdown
发现模糊文件名：
❌ 临时.md

正确处理：
1. 停止操作，不继续使用
2. 询问："这个文件是什么内容？"
3. 根据回答重命名：
   ✅ 2025-11-25_功能需求分析.md
   ✅ 用户认证模块设计方案.md
4. 更新所有引用此文件的链接
```

### 1. 记录类文档 = 日期在前
```

```

```
✅ YYYY-MM-DD

```

```

```

### 5. 分隔符统一
```

✅ 日期与描述：_
✅ 时间内：-
```
